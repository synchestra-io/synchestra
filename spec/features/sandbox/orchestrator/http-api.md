# HTTP API

## Overview

> **Related documents:** [protocol.md](../agent/README.md) (gRPC service that the API proxies to), [orchestrator.md](README.md) (orchestrator interface), [credentials.md](../agent/credentials.md) (credential management details), [testing.md](../observability/testing.md) (API test specifications), [outstanding-questions.md](../outstanding-questions.md) (open design questions).
>
> **OpenAPI contract:** The formal OpenAPI 3.1 specification lives at [`spec/api/sandbox/`](../../../api/sandbox/README.md).

The sandbox HTTP API is served by `synchestra serve --http` and provides RESTful endpoints for the web application to interact with sandbox containers. All sandbox endpoints are prefixed with `/api/v1/sandbox/`. Admin endpoints are prefixed with `/api/v1/admin/sandbox/`.

The API server acts as a stateless proxy — it authenticates requests, validates access permissions, and forwards operations to the appropriate container via the Container Orchestrator and gRPC. The host holds no session state: any host instance can serve any request for any project. Session state lives exclusively inside the container, and session logs are persisted to disk by the container agent.

This design means:
- **Session reconnection works across client disconnects** — the session keeps running in the container; a new WebSocket replays buffered output.
- **Session reconnection works across host restarts** — any new host instance can route to the same container.
- **Session reconnection works across machine migration** — as long as the container is reachable, any host can proxy to it.

**Location in repo**: `internal/api/sandbox/`

## Authentication & Authorization

All requests require a valid authentication token passed as a Bearer token in the `Authorization` header:

```
Authorization: Bearer <token>
```

Token validation extracts `user_id` from the token claims. The `user_id` is used to look up the caller's access level for the target project in the `sandbox_user_project_access` table.

WebSocket connections accept the token via either:
- `Authorization` header (if the client supports it), or
- `?token=<token>` query parameter (for browser clients that cannot set headers on WebSocket upgrade)

### Access Levels

| Level | Description |
|-------|-------------|
| `read` | View container status, list sessions, read session details, stream logs |
| `read_write` | Everything in `read`, plus execute commands and manage credentials |
| `admin` | Everything in `read_write`, plus destroy containers and access admin endpoints |

### Authorization Matrix

| Endpoint | `read` | `read_write` | `admin` |
|----------|--------|--------------|---------|
| `GET /status` | ✅ | ✅ | ✅ |
| `GET /sessions` | ✅ | ✅ | ✅ |
| `GET /sessions/{session_id}` | ✅ | ✅ | ✅ |
| `WS /sessions/{session_id}/logs` | ✅ | ✅ | ✅ |
| `POST /execute` | ❌ | ✅ | ✅ |
| `POST /credentials` | ❌ | ✅ | ✅ |
| `DELETE /credentials/{identifier}` | ❌ | ✅ | ✅ |
| `DELETE /{project_id}` | ❌ | ❌ | ✅ |
| All admin endpoints | ❌ | ❌ | ✅ |

## Common Response Format

All responses use a consistent JSON envelope.

**Success:**

```json
{
  "data": { ... },
  "meta": {
    "request_id": "req-abc123",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}
```

**Error:**

```json
{
  "error": {
    "code": "RESOURCE_EXHAUSTED",
    "message": "Container disk quota exceeded",
    "details": { ... }
  },
  "meta": {
    "request_id": "req-abc123",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}
```

Notes:
- `meta.request_id` is generated by the host for every request and propagated to the container via gRPC metadata. Use it for cross-layer tracing.
- `meta.timestamp` is the host's wall-clock time when the response was serialized (ISO 8601, UTC).
- `error.details` is an optional object carrying structured context (e.g., which field failed validation, the quota that was exceeded). Its schema varies by error code.

## Error Code Mapping

gRPC status codes returned by the container agent are mapped to HTTP status codes by the host. The `error.code` field in the response body matches the gRPC code name.

| gRPC Code | HTTP Status | `error.code` | Description |
|-----------|-------------|--------------|-------------|
| `OK` (0) | 200 | — | Success |
| `INVALID_ARGUMENT` (3) | 400 | `INVALID_ARGUMENT` | Bad request (empty command, timeout out of range, malformed JSON) |
| `DEADLINE_EXCEEDED` (4) | 504 | `DEADLINE_EXCEEDED` | Command exceeded `timeout_seconds` |
| `NOT_FOUND` (5) | 404 | `NOT_FOUND` | Session or credential not found |
| `PERMISSION_DENIED` (7) | 403 | `PERMISSION_DENIED` | User lacks required access level for this operation |
| `RESOURCE_EXHAUSTED` (8) | 429 | `RESOURCE_EXHAUSTED` | Session limit, disk quota, or memory limit exceeded |
| `FAILED_PRECONDITION` (9) | 412 | `FAILED_PRECONDITION` | Container exists but has not completed initialization (state repo not cloned, agent not ready). Client should NOT retry immediately. |
| `ABORTED` (10) | 409 | `ABORTED` | Command aborted (e.g., killed by signal) |
| `INTERNAL` (13) | 500 | `INTERNAL` | Container or host internal error |
| `UNAVAILABLE` (14) | 503 | `UNAVAILABLE` | Container temporarily unreachable (paused, restarting, network issue). Client SHOULD retry with exponential backoff. |
| `UNAUTHENTICATED` (16) | 401 | `UNAUTHENTICATED` | Invalid or missing authentication token |

Errors raised by the host itself (auth failures, rate limiting, request validation) use the same envelope format and the closest matching error code from the table above.

## Sandbox Endpoints

All sandbox endpoints are scoped to a project: `/api/v1/sandbox/{project_id}/...`

The `{project_id}` path parameter is validated against `^[a-zA-Z0-9_-]+$` (alphanumeric, hyphens, underscores). Invalid project IDs return 400.

### POST /api/v1/sandbox/{project_id}/execute

Execute a command in the project's sandbox container. If the container does not exist, it is auto-provisioned before the command runs (the client sees a single response — provisioning is transparent).

**Required access level:** `read_write`

**Request:**

```json
{
  "command": ["git", "clone", "https://github.com/user/repo.git"],
  "working_dir": "/workspace/repos/my-repo",
  "env": {
    "NODE_ENV": "development"
  },
  "timeout_seconds": 300,
  "credentials": ["github-prod"]
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `command` | `string[]` | Yes | Command to execute. First element is the executable, rest are arguments. Must be non-empty. |
| `working_dir` | `string` | No | Working directory inside the container. Default: `/workspace/{project_id}/repos/` |
| `env` | `map<string, string>` | No | Additional environment variables injected for this command only. |
| `timeout_seconds` | `int` | No | Command timeout in seconds. Default: `1800` (30 min). Range: `[1, 86400]`. |
| `credentials` | `string[]` | No | Credential identifiers to inject for this command. Each must already be stored via `POST /credentials`. |

**Response (201 Created):**

```json
{
  "data": {
    "session_id": "sess-abc123",
    "status": "running",
    "stream_url": "/api/v1/sandbox/{project_id}/sessions/sess-abc123/logs"
  },
  "meta": {
    "request_id": "req-def456",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}
```

The response returns immediately after the container accepts the command and assigns a `session_id`. The command runs asynchronously. Clients should connect to the WebSocket `stream_url` to receive real-time stdout/stderr output and the exit event.

**Errors:**

| HTTP Status | Condition |
|-------------|-----------|
| 400 | Empty `command` array, `timeout_seconds` out of range, invalid `working_dir` |
| 403 | User lacks `read_write` access to the project |
| 404 | Referenced credential identifier not found |
| 429 | Rate limit exceeded or container session limit reached |
| 503 | Container unavailable (paused, restarting) |
| 504 | Container provisioning timed out |

### WebSocket /api/v1/sandbox/{project_id}/sessions/{session_id}/logs

Real-time log streaming via WebSocket. Supports both initial connection (during command execution) and reconnection (after client disconnect or host restart).

**Required access level:** `read`

**Connection:**

Upgrade from HTTP to WebSocket. Authentication via:
- `Authorization: Bearer <token>` header, or
- `?token=<token>` query parameter

**Query Parameters:**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `token` | `string` | No | Bearer token (alternative to `Authorization` header). |
| `since` | `string` | No | ISO 8601 timestamp. Skip replay and stream only output produced after this time. If omitted, replay from session start. |

**Messages (server → client):**

All messages are JSON-encoded text frames.

Standard output:
```json
{"type": "stdout", "data": "Installing packages...\n", "timestamp": "2024-01-15T10:30:01Z"}
```

Standard error:
```json
{"type": "stderr", "data": "Warning: deprecated package\n", "timestamp": "2024-01-15T10:30:02Z"}
```

Session exit:
```json
{"type": "exit", "exit_code": 0, "timestamp": "2024-01-15T10:30:05Z"}
```

| Field | Type | Description |
|-------|------|-------------|
| `type` | `string` | Message type: `stdout`, `stderr`, or `exit`. |
| `data` | `string` | Output data. Present for `stdout` and `stderr` types. |
| `exit_code` | `int` | Process exit code. Present only for `exit` type. |
| `timestamp` | `string` | ISO 8601 timestamp of when the output was produced (container clock). |

**Reconnection Behavior:**

The container agent persists session logs to disk. Reconnection replays from the persisted log — no state is required on the host.

| Scenario | Behavior |
|----------|----------|
| First connection, session running | Replay all buffered output from session start, then stream live. |
| Reconnect, session still running | Replay from session start (or from `?since` timestamp), then stream live. |
| Connect to completed session | Send all output + `exit` message, then close the WebSocket. |
| `?since` provided | Skip output before the given timestamp. Useful for resuming after a brief disconnect without replaying the full history. |

**Connection Lifecycle:**

1. **Client connects** — WebSocket upgrade succeeds, server begins sending messages.
2. **Client disconnects** — WebSocket closes, but the session continues running in the container. No output is lost.
3. **Client reconnects** — New WebSocket connection. Buffered output is replayed (respecting `?since` if provided), then live streaming resumes.
4. **Session completes** — Server sends the `exit` message, then closes the WebSocket with close code `1000` (normal closure).

**Server-side keepalive:** The server sends WebSocket ping frames every 30 seconds. If the client does not respond with a pong within 10 seconds, the connection is considered dead and closed.

### POST /api/v1/sandbox/{project_id}/credentials

Store or update an encrypted credential for use in sandbox commands.

**Required access level:** `read_write`

**Request:**

```json
{
  "identifier": "github-prod",
  "credential_type": "git_token",
  "credential_value": "ghp_xxxxxxxxxxxx",
  "description": "GitHub production PAT",
  "expires_at": "2024-12-31T23:59:59Z"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `identifier` | `string` | Yes | Unique name for this credential within the project. Must match `^[a-zA-Z0-9_-]+$`. |
| `credential_type` | `string` | Yes | Type of credential: `git_token`, `ssh_key`, `api_key`, `env_var`. |
| `credential_value` | `string` | Yes | The secret value. Transmitted over TLS, encrypted at rest, **never returned in any response, never logged by the host**. |
| `description` | `string` | No | Human-readable description for UI display. |
| `expires_at` | `string` | No | ISO 8601 expiration timestamp. `null` for non-expiring credentials. |

If a credential with the same `identifier` already exists for this project, it is overwritten (upsert semantics).

**Response (200 OK):**

```json
{
  "data": {
    "success": true,
    "message": "Credential 'github-prod' stored successfully",
    "identifier": "github-prod",
    "credential_type": "git_token"
  },
  "meta": {
    "request_id": "req-pqr678",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}
```

**Errors:**

| HTTP Status | Condition |
|-------------|-----------|
| 400 | Missing required field, invalid `identifier` format, unknown `credential_type` |
| 403 | User lacks `read_write` access to the project |

### DELETE /api/v1/sandbox/{project_id}

**Required access level:** `admin`

**Query Parameters:**

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `clear_cache` | `bool` | No | If `true`, also remove the cached workspace data at `{WORKSPACE_ROOT}/{project_id}/`. Default: `false`. |

**Response (200 OK):**

```json
{
  "data": {
    "message": "Container for project 'proj-123' destroyed",

  "meta": {

```

If the container does not exist, the endpoint returns 200 with a message indicating no container was found (idempotent).

### POST /api/v1/admin/sandbox/{project_id}/stop

Force-stop a container. Sends `SIGTERM` to the container process; if the container does not exit within 10 seconds, sends `SIGKILL`.

All running sessions are terminated. Clients connected via WebSocket receive an `exit` message with `exit_code: -1` before the connection is closed.

**Response (200 OK):**

```json
{
  "data": {
    "message": "Container for project 'proj-123' stopped",
    "previous_status": "running",
    "sessions_terminated": 2
  },
  "meta": {
    "request_id": "req-bcd890",
    "timestamp": "2024-01-15T10:30:00Z"
  }
}
```

**Errors:**

| HTTP Status | Condition |
|-------------|-----------|
| 404 | No container exists for this project |

---

### POST /api/v1/admin/sandbox/{project_id}/restart

Force-restart a container. Equivalent to stop (SIGTERM → SIGKILL) followed by an immediate start.

```json
  "data": {
```

### POST /api/v1/admin/sandbox/{project_id}/evict

Evict a container from the active pool. Stops the container and removes it from the pool, but preserves the workspace cache on disk. Used by LRU eviction logic or manual operator cleanup.

**Response (200 OK):**


### PATCH /api/v1/admin/sandbox/{project_id}/config

Update resource limits for a project's container. Changes take effect on the next container restart — they do not affect the currently running container.

**Request:**

```json
{
  "memory_limit_mb": 1024,
  "cpu_limit": 4.0,
  "disk_quota_gb": 100
}
```

All fields are optional; only provided fields are updated.

| Field | Type | Description |
|-------|------|-------------|

## Rate Limiting

Rate limits are applied per user (identified by `user_id` from the auth token) and enforced by the host before requests are forwarded to containers.

| Category | Limit | Scope |
|----------|-------|-------|
| Execute command (`POST /execute`) | 10 requests/minute | Per user, per project |
| Credential operations (`POST /credentials`, `DELETE /credentials/{id}`) | 30 requests/minute | Per user |
| Status and session queries (`GET /status`, `GET /sessions`, `GET /sessions/{id}`) | 60 requests/minute | Per user |
| Admin endpoints | 30 requests/minute | Per admin user |

**Rate limit headers** are included in every response:

| Header | Description |
|--------|-------------|
| `X-RateLimit-Limit` | Maximum requests allowed in the current window. |
| `X-RateLimit-Remaining` | Requests remaining in the current window. |
| `X-RateLimit-Reset` | Unix timestamp (seconds) when the current window resets. |

When the rate limit is exceeded, the server returns:

- HTTP status: `429 Too Many Requests`
- `Retry-After` header with seconds until the window resets
- Standard error envelope with `error.code: "RESOURCE_EXHAUSTED"`
